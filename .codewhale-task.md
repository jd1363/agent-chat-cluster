## 功能要求

### 1. 输入

### 2. CLI

支持：

```bash
python scripts/scheduler_tick.py --dry-run
python scripts/scheduler_tick.py --dry-run --json
python scripts/scheduler_tick.py --state state/system_state.json --dry-run
python scripts/scheduler_tick.py --write-event --dry-run
```

语义：

- 当前版本必须要求 `--dry-run`。如果不传 `--dry-run`，拒绝执行并 exit 1，说明当前只支持 dry-run，不会真实派工。
- `--json` 输出纯 JSON，stdout 不混入摘要；建议使用 `ensure_ascii=True`，避免 Windows 编码问题。
- 非 `--json` 输出人类可读摘要。
- `--write-event --dry-run`：允许写入一个调度决策事件到 Event Layer，事件类型建议 `scheduler.tick.evaluated`。注意仍然不派工、不发消息、不改 tasks。

### 3. 调度逻辑

第一版只做保守判断：

从 state 中读取：
- `tasks.items`
- `agents.items`
- `policy.maxConcurrency`
- `scheduler.pendingTasks`
- `scheduler.inProgressTasks`
- `scheduler.availableAgents`
- `scheduler.backpressure`

调度步骤：

1. 找出 `status == "pending"` 的任务。
2. 找出 `enabled == true` 的 Agent。
3. 排除 disabled Agent。
4. 判断 maxConcurrency：
   - 如果 `inProgressTasks >= maxConcurrency`，整体 blocked，reason 包含 `maxConcurrency reached`。
5. 判断是否有 pending task：
   - 没有 pending，则 decision 为 `idle`。
6. 判断是否有可用 Agent：
   - 没有 enabled Agent，则 decision 为 `blocked`。
7. 如果可以调度：
   - 选择一个 pending task。
   - 选择一个 enabled Agent。
   - 只输出建议，不修改 tasks，不发消息。

### 4. 任务选择策略

简化优先级：

- priority 顺序：high > medium > low > 其他
- 同优先级按 `createdAt` 升序

### 5. Agent 选择策略

简化策略：

- 只从 `enabled == true` 的 agents 中选
- 优先选择当前任务数较少的 Agent，可使用 `tasks.byAssignee` 作为 load
- 同负载按 agent id 排序
- disabled Agent 绝不允许被选中

- 不修改任务台账。
